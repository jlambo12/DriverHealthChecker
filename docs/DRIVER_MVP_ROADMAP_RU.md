## 1) Цель MVP
Сделать стабильный и понятный для пользователя сценарий:
1. Приложение сканирует важные драйверы.
2. Показывает только релевантный список (без "шума").
3. Даёт ясный статус по каждому драйверу и действие "что делать дальше".
4. Поддерживает повторное сканирование и фиксирует изменения версий.

---

## 2) Определение MVP (что обязательно должно быть)

### 2.1 Категории, которые должны покрываться
- GPU
- Network (Ethernet / Wi‑Fi / Bluetooth)
- Storage (NVMe/SATA/RAID/VMD и т.п.)
- Audio Main (встроенное аудио)
- Audio External (внешние аудиоинтерфейсы/аудиокарты)

### 2.2 Что должен видеть пользователь
- Название устройства
- Категория
- Производитель
- Версия драйвера
- Дата драйвера
- Статус (Актуален / Стоит проверить / Требует внимания / Недавно обновлён)
- Кнопка перехода к официальному источнику обновления

### 2.3 Критерии качества списка
- Минимум ложноположительных системных/виртуальных устройств.
- В списке остаются только "пользовательски значимые" драйверы.
- Стабильный порядок и детерминированный выбор "лучшей" записи в категории.

## 3) Текущее состояние и пробелы

### Уже есть
- Сканирование Win32_PnPSignedDriver.
- Базовая категоризация по правилам и blacklist.
- Базовая оценка статусов по дате.
- Официальные действия (URL / app / поиск).
- Повторное сканирование и пометка "Недавно обновлён".

### Пробелы до завершённого MVP
- Критичных пробелов для MVP не осталось.
- Остались только post-MVP улучшения (телеметрия качества, OEM-специализация правил).

## 4) Roadmap по этапам

## Этап A — Нормализация правил и структуры (1–2 дня)
**Цель:** сделать правила расширяемыми и прозрачными.

### Задачи
- Вынести все словари/термины категоризации в `DriverRules` и сгруппировать по категориям.
- Добавить комментарии к правилам: зачем правило и какие устройства покрывает.
- Разделить blacklist на:
  - системный шум,
  - виртуальные устройства,
  - служебные компоненты.

### Результат этапа
- Единая точка управления правилами.
- Понятно, как добавлять новые модели/бренды без изменения UI-логики.

---

## Этап B — Эталонные наборы и проверка точности (2–4 дня)
**Цель:** подтвердить качество классификации на реальных данных.

### Задачи
- Собрать 10–20 анонимизированных снэпшотов драйверов с разных типов ПК:
  - Intel + NVIDIA,
  - AMD + AMD,
  - Intel iGPU only,
  - ноутбуки с OEM-драйверами,
  - ПК с внешними аудиоинтерфейсами.
- Добавить тестовые fixtures (JSON/CSV) и ожидаемые категории.
- Сделать проверку классификатора на fixtures (автотест/консольная проверка).
- Использовать `scripts/generate-fixtures-from-reports.ps1` для ускоренной генерации шаблона fixtures из `scan-history`.

### KPI этапа
- Точность по категориям (ориентир):
  - GPU: >= 99%
  - Network: >= 95%
  - Storage: >= 95%
  - Audio Main: >= 95%
  - Audio External: >= 90%
- Статус: ✅ выполняется на текущем наборе fixture-тестов и KPI-тестов.

---

## Этап C — UX списка драйверов для пользователя (2–3 дня)
**Цель:** улучшить читаемость и доверие к результатам.

### Задачи
- Добавить явный признак "Почему в списке" (например: категория + ключевой матч).
- Добавить фильтры по категориям и по статусу.
- Добавить "Показать скрытые/системные" как отдельный диагностический режим (по умолчанию выключен).
- Уточнить текст статусов и подсказок действий.

### Результат этапа
- Пользователь понимает, почему именно эти драйверы отображены.
- Меньше вопросов "куда делся мой драйвер" и "почему это важно".

---

## Этап D — Полировка сценария обновлений (1–2 дня)
**Цель:** финализировать MVP-поток "проверил → обновил → перепроверил".

### Задачи
- Убедиться, что после обновления драйвера статус/версия корректно изменяются на рескане.
- Добавить заметку в UI: обновления делаются только из официальных источников.
- Проверить edge-cases (нет сети, недоступный URL, OEM утилиты без установленного клиента).

### Результат этапа
- Предсказуемый end-to-end сценарий для конечного пользователя.

---

## 5) Definition of Done для MVP
MVP по драйверам считается завершённым, если:
1. Категоризация стабильна на эталонных наборах и соответствует KPI.
2. Список драйверов понятен пользователю (минимум шума, понятные действия).
3. Рескан корректно показывает изменения версий/дат.
4. Нет изменений, ломающих текущий релизный поток.
5. Документация по правилам и поддержке списка драйверов обновлена.

### Текущий статус DoD
- ✅ Категоризация закреплена через fixtures + KPI-тесты.
- ✅ UX списка дополнен причинами, фильтрами и диагностическим режимом.
- ✅ Рескан и отчёты поддерживают проверку изменений версий/дат.
- ✅ Изменения покрыты unit-тестами и не ломают текущий релизный поток.
- ✅ Документация roadmap актуализирована.

## 6) Приоритеты задач (P0 / P1 / P2)

### P0 (обязательно до MVP)
- Эталонные fixtures + проверка качества классификации.
- Детерминированные правила отбора "лучшего" драйвера в категории.
- Прозрачная причина отображения драйвера в списке.

### P1 (сильно желательно)
- Диагностический режим "показать скрытые".
- Расширенные подсказки по официальным источникам.

### P2 (после MVP)
- Телеметрия качества классификации (опционально).
- Профили правил по OEM-вендорам.

## 7) Предлагаемый порядок внедрения
1. Этап A (рефактор правил).
2. Этап B (fixtures и KPI).
3. Этап C (UI-улучшения списка).
4. Этап D (полировка end-to-end).

Итог: сначала качество данных, затем UX, затем полировка.
