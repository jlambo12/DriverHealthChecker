# Безопасный workflow: как работать через PR и не ломать проект

Ниже — короткий, практичный процесс. Я буду вести тебя именно так в следующих шагах.

## 0) Базовые правила

1. Любая задача делается в **отдельной ветке**.
2. Один PR = одна небольшая цель (не смешивать рефакторинг и фичу).
3. Перед merge обязательно: `restore/build/tests` + ручная проверка ключевого сценария.
4. Merge только в "зелёном" состоянии (без красных проверок).

## 1) Создать ветку под задачу

Сначала определяем базовую ветку (в этом окружении часто это `work`, а не `main`):

```bash
git branch --show-current
```

```bash
git checkout work
git checkout -b feat/phase0-logging
```

Правило именования:
- `feat/...` — новая функциональность,
- `fix/...` — исправление,
- `refactor/...` — рефакторинг,
- `docs/...` — документация.

## 2) Работать маленькими шагами

- Сделал 1 логическое изменение → сразу локальная проверка.
- Затем commit с понятным сообщением.

Пример:
```bash
git add DriverHealthChecker.App/AppLogger.cs
git commit -m "refactor: add app logger"
```

## 3) Локальный checklist перед PR

> Для этого проекта запускать на Windows (WPF).

```bash
dotnet restore DriverHealthChecker.sln
dotnet build DriverHealthChecker.sln -c Release
dotnet test
```

Если тестов пока нет, значит обязательно:
- открыть приложение,
- проверить сканирование,
- проверить повторное сканирование,
- проверить кнопку открытия источника.

## 4) Подготовить PR

В PR всегда пишем:
1. **Что сделано** (файлы/логика).
2. **Зачем** это изменение.
3. **Как проверено** (точные команды + результат).
4. **Риски/ограничения**.

## 5) Обновить ветку перед merge

```bash
git checkout work
git checkout feat/phase0-logging
git rebase work
```

Если есть конфликты:
- решить,
- снова прогнать `build/tests`.

## 6) Merge без поломок

Рекомендуемый режим: **Squash and merge** (чистая история).

Перед нажатием merge проверяем:
- CI зелёный,
- нет нерешённых комментариев,
- есть план отката.

## 7) План отката (на случай проблем)

Минимум:
- сохранить hash merge-коммита,
- при необходимости сделать `revert` отдельным PR,
- не пушить “горячие” правки напрямую в `main`.

## 8) Что я буду делать дальше (чёткий формат)

В каждом моём следующем шаге ты получишь:
1. **Шаг сейчас** (например: "Phase 1.1 — выделяю `IDriverStatusEvaluator`").
2. **Изменяемые файлы**.
3. **Команды проверки**.
4. **Критерий готовности** (что считается done).
5. **Когда открываем PR и когда merge**.

## 9) Как лучше работать: один PR или много маленьких

Короткий ответ: **лучше много маленьких PR**, а не один бесконечный.

### Когда обновляем текущий PR (тот же branch)

Оставляем один и тот же PR, если:
- это продолжение **той же самой цели**,
- изменения логически связаны,
- объём всё ещё небольшой и ревью читается.

Пример: в PR \"Phase 0 logging\" сначала добавили логгер, потом в этой же ветке
добавили 2–3 недостающих вызова логирования.

### Когда создаём новую ветку и новый PR

Создаём новый PR, если:
- началась **другая цель** (например, уже не логирование, а MVVM-рефакторинг),
- изменения большие и смешиваются с предыдущими,
- текущий PR уже готов к merge и не нужно его раздувать.

Пример: после merge PR с логированием открываем новый `refactor/...` PR
на выделение `IDriverStatusEvaluator`.

### Рекомендуемая схема работы с мной

1. Я даю тебе **план следующего маленького PR** (1 цель, 1 результат).
2. Делаем изменения в отдельной ветке.
3. Проверяем локально (`restore/build/test` + ручной smoke-check).
4. Открываем PR и доводим до merge.
5. После merge — **новая ветка и новый PR** для следующего шага roadmap.

Это самый безопасный режим: минимальный риск поломок и быстрый откат при проблеме.
