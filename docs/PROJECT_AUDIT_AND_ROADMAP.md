# DriverHealthChecker: аудит, проверка и roadmap

## 1) Что проверено

- Структура решения (`DriverHealthChecker.sln`, WPF-проект `DriverHealthChecker.App`).
- Конфигурация проекта (`TargetFramework=net10.0-windows`, `UseWPF=true`).
- Основная логика сканирования, категоризации и статусов драйверов (`MainWindow.xaml.cs`).
- Логика автообновления через Velopack (`App.xaml.cs`).

## 2) Выводы по текущему состоянию

### Сильные стороны

1. Есть рабочий MVP-поток:
   - сканирование,
   - повторное сканирование,
   - визуализация статусов,
   - переход к официальным источникам.
2. Реализована полезная функция "Недавно обновлён" через сравнение снимков.
3. UI уже пригоден для демо: читаемые цвета и базовая группировка данных.

### Ключевые риски и технический долг

1. **Слишком много ответственности в `MainWindow.xaml.cs`** (поиск, фильтрация, бизнес-правила, действия UI).
2. **Много `catch { }` без логирования** — ошибки теряются, сложно диагностировать баги.
3. **Жестко зашитые строки/эвристики** (категории, blacklist, URL) прямо в коде.
4. **Нет автоматических тестов** на правила категоризации и вычисление статусов.
5. **Сканирование выполняется в UI-потоке**; на слабых ПК возможна "заморозка" окна.
6. **Платформенная привязка к Windows/WMI** без абстракций, что усложняет развитие и тестируемость.

## 3) Что уже добавлено для окружения

Добавлен скрипт `scripts/setup-dev-env.ps1`, который:

- проверяет наличие .NET SDK 10,
- при необходимости ставит SDK через официальный `dotnet-install.ps1`,
- валидирует target framework,
- выполняет `dotnet restore` по решению.

## Phase 1 — Рефакторинг архитектуры (2 недели)

- Разделить `MainWindow.xaml.cs` на сервисы:
  - `IDriverScanner` (доступ к WMI),
  - `IDriverClassifier` (категоризация),
  - `IDriverStatusEvaluator` (статусы),
  - `IOfficialActionResolver` (официальные источники).
- Подготовить переход к MVVM (минимум ViewModel для таблицы и summary).

**Результат:** код проще сопровождать и тестировать.

## Phase 2 — Тестирование логики (2 недели)

- Добавить unit-тесты для:
  - `IsGpu/IsNetwork/IsStorage/IsMainAudio/IsExternalAudio`,
  - `EvaluateStatus`,
  - выбора "лучшего" драйвера по дате/версии.
- Добавить smoke-тест на загрузку окна.

**Результат:** безопасный рефакторинг и контроль регрессий.

## Phase 3 — UX и производительность (2 недели)

- Выполнить сканирование асинхронно с индикатором прогресса.
- Добавить "последняя успешная ошибка/причина" в UI.
- Добавить фильтры/поиск по таблице.

**Результат:** улучшение отклика и пользовательского опыта.

## Phase 4 — Надежный релизный контур (1–2 недели)

- Зафиксировать версию SDK через `global.json`.
- Настроить CI (restore + build + test).
- Формализовать changelog и версионирование.

**Результат:** предсказуемые релизы и меньше ручных ошибок.

## 5) Definition of Done для ближайшего milestone

- Есть логирование и нет пустых `catch`.
- Бизнес-логика вынесена из code-behind в сервисы.
- Покрытие ключевых правил тестами не ниже 70%.
- В CI проходят `restore/build/test`.
- Обновлен README с шагами запуска и ограничениями платформы.
